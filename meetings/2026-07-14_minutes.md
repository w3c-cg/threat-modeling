### User Mediation, Prompt API, and Exfiltration

* Tom Jones: I am concerned about interaction that starts before the user is
  aware of it. A page could load JavaScript and initiate agentic activity while
  the user is still reading the first page.

* Joe Andrieu: I recorded this as unapproved AI interaction. I also recorded
  unapproved tools being registered with the agent; I expect some user prompt
  or approval mechanism to address that risk.

* Dominic Farolino: I do not see a formal Prompt API and WebMCP integration in
  the scenario we are discussing. A page can connect a model to its own
  JavaScript functions without WebMCP, so that broader threat does not depend
  on WebMCP.

* Tom Jones: Exfiltration should still be listed. The question is whether
  WebMCP makes it easier and, if it does not, that can be recorded as part of
  the response.

* Joe Andrieu: The fact that an attack can also occur through another channel
  is not a reason to omit it. We should record the threat and then identify
  whether WebMCP, Prompt API, an agent, or another component owns the response.

* Dominic Farolino: A better framing is a non-user-mediated model interacting
  with parts of a page. That model could come from Prompt API, downloaded
  JavaScript, or a remote service; the threat is not specific to one of those
  mechanisms.

* Joe Andrieu: I support that framing.

* Tom Jones: That captures my concern: the interaction is not mediated by the
  user.

* Dominic Farolino: I believe Prompt API is gated by Permissions Policy, but I
  do not recall a user-activation requirement in the specification.

* Simone Onofri: User mediation is therefore an important part of the model,
  even where the concern belongs to agentic browsing rather than WebMCP alone.

* Tzviya Siegman: Because this session is specifically about WebMCP rather
  than generative AI in general, the documentation should make that scope
  clear.

### Repudiation, Visibility, and Model Scope

* Joe Andrieu: If an agent takes an action that legally or financially binds
  the user, the user may later deny that the action came from them. I want
  visibility, such as a log of what the agent sent over the WebMCP channel.

* Julia Pagnucco: I am not sure this is new to WebMCP because UI actuation has
  a similar problem, but I agree that repudiation is a threat.

* Simone Onofri: Watching an agent's actions can help a user understand and
  interrupt what it is doing, and it may also support accountability. If
  WebMCP can provide better logging than unstructured agentic browsing, that
  could be a security benefit of the standard.

* Tom Jones: We should not examine WebMCP only in isolation because it is used
  with an agent. The embodied deployment is needed to understand the actual
  threats.

* Joe Andrieu: The model needs that embodiment. Abstract technology still has
  concrete ingestion, execution, and output paths where threats appear.

### Tool Trust, Integrity, and Cross-Origin Behavior

* Gianluca Sassetti: A site can expose a malicious or unverified tool, and its output
  may be processed outside the site's sandboxed context. I see a threat where
  neither the exposed tool nor its output has an adequate trust or verification
  mechanism.

* Simone Onofri: A website is already an untrusted party in the web model and
  runs across a trust boundary. We should still record the more specific
  concern about unverified tools and outputs.

* Gianluca Sassetti: I also see a tampering concern if the stored tool metadata or
  manifest information has no integrity check.

* Joe Andrieu: An agent may have access to tools associated with different
  sites or tabs. A malicious site could influence the agent and cause it to
  return information obtained from another site.

* Simone Onofri: The discussion indicated that tools are same-origin by
  default and that cross-origin exposure requires explicit opt-in. That is a
  relevant existing control, but it does not remove every cross-site agent
  concern.

* Dominic Farolino: The tools are scoped to a tab in the scenario we are
  discussing, but an agent may still interact with multiple tabs and carry
  information between them.

* Joe Andrieu: That is the concern. The agent is a non-deterministic process
  that may access several open tabs and communicate with tools available from
  different origins.

* Dominic Farolino: Cross-contamination between sites is a valid threat, but
  we have treated much of it as part of the agent security model. An agent can
  move information between sites without WebMCP; WebMCP makes interaction with
  a site easier and more structured.

* Julia Pagnucco: The complete system has at least three security layers:
  API-level features, agent security, and site security. The API cannot carry
  all of those responsibilities by itself.

* elf Pavlik: If we classify a threat as broader than WebMCP, we should still
  identify which threat model or work item will own it. Otherwise, it may be
  dropped because nobody is responsible for documenting it.

* Joe Andrieu: We should enumerate the threats even when another specification
  or component owns the response. A specification can acknowledge a threat and
  point readers to the relevant control or dependency.

* Dominic Farolino: A broader threat model for agents acting on web content
  would be useful. It could cover concerns that apply whether or not the agent
  uses WebMCP or Prompt API.

### Threats Elicited During the STRIDE Exercise

* elf Pavlik: A user could be tricked into installing a malicious agent
  extension with broad permissions. This may be a general browser-extension
  threat, with WebMCP providing another reason for a user to install such an
  extension.

* Simone Onofri: That is a valid threat, although it may belong in the Threat
  Model for the Web or browser-extension work rather than only in the WebMCP
  model.

* Simone Onofri: Tool names, descriptions, schemas, or other metadata could
  carry prompt injection. Limiting metadata length may reduce some payloads,
  but it does not by itself resolve the threat.

* Simone Onofri: A tool result can also contain poisoned or untrusted content
  that influences a later request. An untrusted-content label may help the
  agent handle that result more carefully.

* Simone Onofri: A malicious page could dynamically register many tools or
  repeatedly change them, exhausting browser, agent, or model resources and
  potentially creating a denial-of-service or crash surface.

* Simone Onofri: Automatic submission and unintended tool actions need
  attention, particularly when an agent selects or submits files from the
  user's system.

* Simone Onofri: Private browsing may not protect the user if the agent or its
  provider carries context from earlier activity into the private session.
  The model should show where that retained context lives.

* Simone Onofri: A same-origin document may also try to induce the agent to
  open another context and perform an unrelated action. We should record that
  path even if existing web controls address part of it.

* Zahra Ebadi: In a browser-extension deployment, a malicious extension could
  impersonate a user-approved agent and call tools as if it were that agent.

* Simone Onofri: Agent impersonation makes sense as a threat for that specific
  deployment scenario.

### Human in the Loop, Consent, and Trust

* Nishant Jain: MCP elicitation can let a server ask the client to confirm a
  cross-boundary action. Applying the same idea to WebMCP is complicated
  because the web page and browser boundaries differ, but I will develop a
  proposal for how browser guardrails or elicitation could work here.

* Giovanni Corti: I want to know whether the tool or the agent decides when a
  human must be brought into the loop, for example before confirming a
  payment. I would treat both the tool and the agent as untrusted.

* Julia Pagnucco: In the declarative design discussed here, disabling automatic
  submission requires human involvement before a form is submitted. In the
  imperative design, a tool can request user interaction, but the agent must
  still act on that request. We are considering how to make this more
  verifiable.

* Giovanni Corti: That places responsibility on both sides. The open question
  is how the tool and agent are designed to request and honor user interaction.

* Julia Pagnucco: We are weighing whether the site or the agent should obtain
  consent. It is a difficult design question because we are working from a
  zero-trust position.

* elf Pavlik: A user may need to trust an agent more than a website because the
  agent can operate across multiple sites and origins. A site may also try to
  poison that agent.

* Julia Pagnucco: We use a zero-trust model because page content can prompt
  inject an agent and change its behavior. We should not assume either side is
  harmless.

* Simone Onofri: A user chooses and therefore has to place some trust in their
  browser or agent as a user agent acting in their interest. At the same time,
  software components should continue to treat user and page input as
  untrusted.

* Giovanni Corti: The average user may not perform due diligence on an agent,
  just as they may not carefully assess browser extensions. We may have to rely
  on the agent, but that does not mean we should treat it as inherently
  trustworthy.

* Joe Andrieu: An agent can take actions that are indistinguishable from direct
  user actions. Because WebMCP provides a distinct channel, a website may be
  able to recognize an agent-originated action and apply stronger consent or
  authority checks.

* Joe Andrieu: Browsers use a user click to grant some JavaScript capabilities.
  We may need an analogous signal for WebMCP, although I do not yet know what a
  reliable human-in-the-loop mechanism would look like.

* Simone Onofri: I want to test whether agent-mediated interaction changes the
  assumptions behind familiar attacks such as request forgery and clickjacking,
  particularly where a target site already has a script-injection weakness.
  That broader automation risk is one reason to model agentic web browsing as
  well as WebMCP itself.

* Simone Onofri: WebMCP is intended as a structured way for an agent to
  interact with a website, not as a delegation protocol. The structure may
  make tool selection more reliable than visually choosing a button, but the
  broader change to web interaction still needs threat analysis.

* elf Pavlik: In terms of capability and required trust, the agent appears
  closer to a browser or user agent than to an OAuth client or ordinary
  application.
